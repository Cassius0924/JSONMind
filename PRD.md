1. 产品概述

1.1. 背景与痛点
在软件开发与数据交互中，JSON 是最普遍的数据格式。然而，随着业务复杂度增加，JSON 文件往往变得极其庞大且嵌套极深。阅读、调试和手动编辑原始 JSON 文本非常痛苦，容易出错且难以把握整体结构。

1.2. 产品目标
开发一款工具，能够将任意 JSON 数据实时转换为可视化的“思维导图”结构。用户不仅能清晰地查看数据层级，还能直接在导图上进行编辑（增删改查），并最终将编辑后的结果无损导出为标准的 JSON 格式。

1.3. 核心价值主张
“所见即所得”的结构化 JSON 编辑体验。将抽象的代码文本转换为直观的拓扑图形。

1.4. 目标用户

* 后端工程师： 调试复杂的 API 响应或配置文件。
* 前端工程师： 理解和Mock后端返回的数据结构。
* 测试工程师： 快速构造不同结构的测试数据。
* 数据分析师： 快速浏览陌生数据集的Schema。

1. 用户故事 (User Stories)

* [US-01] 作为一名开发者，我希望将一段复杂的 JSON 文本粘贴到工具中，立即看到一个清晰的思维导图，以便我理解其结构。
* [US-02] 作为一名开发者，我希望在思维导图中快速找到一个深层嵌套的字段并修改它的值，而不需要在文本编辑器中痛苦地滚动和查找。
* [US-03] 作为一名测试人员，我希望在一个数组节点下快速添加几个新的对象元素，用于构造测试用例，并确认导出后的 JSON 格式正确。
* [US-04] 作为一名用户，我希望一眼就能看出哪些字段是字符串，哪些是数字或布尔值，通过颜色和图标进行区分。

1. 功能需求说明 (Functional Requirements)

3.1. 输入与导入模块

* F-IN-1: 文本输入区： 提供一个代码编辑器区域（如 Monaco Editor），支持语法高亮，用于粘贴或输入原始 JSON 文本。
* F-IN-2: 实时校验： 输入过程中实时检查 JSON 格式的合法性。如果格式错误，给予明显的提示，且不进行导图渲染。
* F-IN-3: 格式化按钮： 提供按钮一键美化（Pretty Print）输入区中的 JSON 文本。

3.2. 核心可视化渲染引擎 (关键需求)
基于确认的视觉效果，渲染引擎需遵循以下规则将 JSON 映射为导图：
3.2.1. 布局与结构规则

* R-LAYOUT-1: 水平布局： 根节点在最左侧，子节点向右侧延展。整体数据流方向为从左到右 (Left-to-Right)。
* R-LAYOUT-2: 对称分布： 每个父节点的直接子节点在垂直方向上应保持对称、均衡分布，避免头重脚轻。

3.2.2. 节点映射逻辑规则
JSON 数据类型分为容器型（Object, Array）和基本值型（String, Number, Boolean, Null）。

* R-NODE-1: 根节点渲染： JSON 的最外层（通常是 {} 或 []）作为思维导图的 Root 节点。
* R-NODE-2: 容器型节点渲染 (Object/Array)：
  * 作为“父节点”存在，必定产生分支连接其内部元素。
  * 节点内容仅展示 Key 名称（如果是数组元素， Key 为索引 [0], [1]）。
  * 必须包含类型图标提示（如 {} 或 []）。
* R-NODE-3: 基本值型节点渲染 (Primitive Types) - 核心设计：
  * 当一个 Key 对应的值是基本类型时，不再产生新的子分支。
  * 采用**“组合式分裂节点 (Combined Split Node)”**设计：
    * 节点左半部分（灰色底）：显示 Key 名称。
    * 节点右半部分（彩色底）：显示具体 Value 值。
  * 左右部分在视觉上是一个整体，但有明显分割线。

3.2.3. 视觉样式编码规则
为提高识别效率，对不同数据类型应用固定的颜色和图标方案（应用于 R-NODE-3 的右半部分及 R-NODE-2 的图标）：

| 数据类型 | 颜色编码 (参考) | 图标/标识 (参考) | 备注 |
| --- | --- | --- | --- |
| String | 绿色 (Green) | abc 小图标 | |
| Number | 蓝色 (Blue) | # 小图标 | |
| Boolean | 橙色 (Orange) | Checkmark/X 图标 | 值显示 true/false |
| Null | 灰色 (Grey) | ∅ 或文字 null | |
| Object | 默认/灰色 | {} 图标 | 仅用于容器节点 |
| Array | 默认/灰色 | [] 图标 | 仅用于容器节点 |

3.3. 交互与编辑模块

思维导图必须是可交互的，所有操作需实时同步回底层 JSON 数据模型。

* F-EDIT-1: 节点值编辑： 双击“组合式分裂节点”的右侧（Value 区域），弹出输入框修改值。
  * 类型自动推断： 输入数字字面量自动转为 Number 类型，输入 "true"/"false" 自动转为 Boolean。同时提供手动切换类型的下拉菜单。
* F-EDIT-2: 节点 Key 编辑： 双击 Object 子节点的 Key 区域（包括容器节点和组合节点的左侧），修改属性名。
  * 约束： Array 的索引 Key ([0]) 不可编辑。
* F-EDIT-3: 新增节点 (Add Child/Sibling)：
  * 选中容器节点 (Object/Array)，提供“添加子节点”按钮。
  * 上下文感知：
    * 若父节点是 Object：新增节点需同时输入 Key 和 Value。
    * 若父节点是 Array：新增节点自动生成下一个索引 Key（如 [n+1]），只需输入 Value。
* F-EDIT-4: 删除节点： 选中任意节点，提供删除功能。删除容器节点会级联删除其所有子节点。
* F-EDIT-5: 拖拽排序（高阶需求）： 允许在 Array 容器内拖拽调整元素的顺序。

3.4. 输出与导出模块

* F-OUT-1: 实时同步： 导图上的任何编辑操作，应立即更新输入区（F-IN-1）的 JSON 文本内容。
* F-OUT-2: 导出文件： 提供按钮将当前导图结构导出为 .json 文件。
* F-OUT-3: 导出图片： 提供按钮将当前思维导图视图导出为 PNG/SVG 图片。

1. 非功能需求 (Non-Functional Requirements)

4.1. 性能要求

* 渲染性能： 对于中等规模的 JSON (例如 500行以内，5层深度)，首次渲染时间应小于 1 秒。
* 大数据量处理： 面对超大 JSON 文件（如 > 2000行），系统不应崩溃。
  * 策略： 默认仅展开前 3 层级，后续层级采用“懒加载”或手动点击展开。考虑使用虚拟滚动技术优化画布性能。

4.2. 易用性要求

* 支持画布缩放 (Zoom in/out) 和平移 (Pan)。
* 提供快捷键支持（例如：Tab 插入子节点，Enter 插入兄弟节点，Delete 删除节点）。

4.3. 数据一致性 (Schema Consistency)

* 这是最关键的技术难点。系统内部必须维护一个严格带有类型信息的数据模型树。
* 确保在编辑过程中，不会意外将数字 123 变成了字符串 "123"，除非用户明确进行了类型转换操作。
